# 02. 사이트맵 & 라우팅

디바이스는 URL prefix로 구분한다. 루트(`/`)는 PC, `/mobile` 이하가 모바일.

## PC (`/`) — 8개 페이지

| # | URL | 페이지 | 근거 | 필수 |
|---|---|---|---|---|
| 1 | `/` | 메인 | Figma pc-main(452:309) 그대로 구현 + 제안서 KEY SCREEN ① | ✅ 필수 |
| 2 | `/login` | 로그인 | RFP FNR-01 (이메일 로그인 + 소셜 버튼 목업) | ✅ 필수 |
| 3 | `/search` | 검색 결과 / 호텔 탐색 | RFP FNR-03, 제안서 KEY SCREEN ② — 목적 필터·회원가 비교 | 선정 |
| 4 | `/hotels/:hotelId` | 호텔·객실 상세 | RFP FNR 4-1, 제안서 KEY SCREEN ③ — 객실 목록·정가/회원가 | 선정 |
| 5 | `/booking` | 예약하기 (옵션 → 혜택 적용 → 결제, 3-step) | RFP FNR-04, 제안서 원스톱 예약 | ✅ 필수 |
| 6 | `/booking/complete` | 예약 완료 | RFP FNR 4-4 | 선정 (예약 플로우 완결) |
| 7 | `/mypage` | 마이페이지 / 멤버십 대시보드 | RFP FNR-05·08, 제안서 KEY SCREEN ④ — 등급 게이지·포인트 소멸·예약 내역(**TanStack Table**) | 선정 |
| 8 | `/offers` | 스페셜 오퍼 (프로모션 목록) | RFP FNR 2-1·3-8·8-10, Figma 메인의 "Promotions & Packages" 확장 | 선정 |
| 9 | `/guide` | 페이지 URL 가이드 | 전체 페이지 목록·URL을 표로 정리한 내부용 인덱스 (스타일링 최소화) | 유틸 |

> 선정 이유 요약: 3~6은 "메인 → 탐색 → 상세 → 예약 → 완료"의 핵심 여정을 끊김 없이 시연하기 위한 최소 구성. 7은 제안서 3축 중 '고객관계'를 보여주는 대표 화면이자 TanStack Table 활용처. 8은 D2C 핵심 전략(공식 채널 전용 혜택)의 시연용.

## 라우팅 규칙

- PC 페이지는 공용 `PcLayout`(Header + Footer), 모바일은 `MobileLayout`(AppBar + BottomNav) 중첩 라우트로 감싼다.
- 모바일 하단 내비게이션: 홈 / 검색 / 예약(CTA 강조) / 혜택 / MY — Figma 하단 탭 구성 따름.
- 존재하지 않는 경로 → 각 디바이스 메인으로 리다이렉트.
- 미로그인 상태에서 혜택 적용 단계 진입 시 로그인 유도 (프로토: 로그인 페이지 이동 후 복귀).

## 사용자 여정 (시연 시나리오)

```
[PC]     메인 → (목적 카드/검색) → 검색 결과 → 호텔 상세 → 예약(옵션→혜택→결제) → 완료 → 마이페이지
[Mobile] 홈 → 검색 결과 → 호텔 상세 → 옵션 추가 → 회원 혜택 적용 → 결제 → 예약 완료
```
